# Rules: Solana Observability Production Monitoring

Auto-loading production rules for Solana observability work. These rules apply to
all generated monitoring code, alert rules, dashboards, runbooks, logs, traces,
health checks, and deployment examples in this repository.

## Operating Philosophy

Monitoring exists to protect users, funds, and protocol reliability. Instrument
before failure, alert on symptoms before causes, and make every page actionable.
A dashboard without thresholds is decoration. An alert without a runbook is noise.
A log line with a secret is a security incident.

Default priority order:

1. User funds and authority safety
2. Transaction success rate and confirmation reliability
3. RPC/indexer/webhook freshness
4. Program correctness and CU headroom
5. Frontend/wallet user experience
6. Capacity, cost, and long-term trend monitoring

## Canonical SLOs and SLIs

| Service | Default SLO | Primary SLI | Error Budget Window |
|---|---:|---|---|
| Transaction success | 99.5% | successful tx / submitted tx | 30d |
| RPC availability | 99.9% | healthy RPC probes / all probes | 30d |
| Health endpoints | 99.99% | successful health checks / checks | 30d |
| Indexer freshness | 99.5% | indexer lag <60s | 30d |
| Frontend load | p95 <3s | browser load duration | 7d |
| Wallet signing flow | 99% | completed sign+confirm / started | 7d |

If a user gives custom SLOs, use them. If not, use these defaults and state them
in every dashboard, alert, and runbook generated by the skill.

## Canonical Metric Contract

Use these metric names unless the target codebase already has a documented schema.
When adapting existing metrics, add compatibility notes instead of silently mixing
names.

| Metric | Type | Required Labels | Purpose |
|---|---|---|---|
| `solana_rpc_healthy` | gauge | `cluster`, `endpoint` | 1 if endpoint passes probe |
| `solana_rpc_request_duration_seconds` | histogram | `cluster`, `endpoint`, `method` | RPC latency distribution |
| `solana_rpc_errors_total` | counter | `cluster`, `endpoint`, `method`, `error_class` | RPC errors |
| `solana_slot_lag_slots` | gauge | `cluster`, `endpoint` | slots behind trusted tip |
| `solana_transaction_total` | counter | `cluster`, `program_id`, `instruction`, `status` | submitted/confirmed tx outcome |
| `solana_transaction_confirmation_seconds` | histogram | `cluster`, `program_id`, `instruction` | confirmation duration |
| `solana_instruction_cu_consumed` | histogram | `cluster`, `program_id`, `instruction` | CU distribution |
| `solana_fee_payer_balance_sol` | gauge | `cluster`, `account_alias` | fee payer runway |
| `solana_indexer_lag_seconds` | gauge | `cluster`, `indexer` | freshness of indexed state |
| `solana_webhook_delivery_total` | counter | `cluster`, `provider`, `status` | webhook delivery health |
| `solana_program_upgrade_detected_total` | counter | `cluster`, `program_id` | upgrade/data hash changes |

### Label Safety

Allowed bounded labels:

- `cluster`
- `endpoint` alias such as `helius-primary`, never raw URL
- `provider`
- `method`
- `program_id`
- `instruction`
- `status`
- `error_class`
- `account_alias`
- `indexer`

Forbidden labels:

- wallet address
- transaction signature
- raw RPC URL
- API key
- user ID
- session ID
- IP address
- full user agent
- unbounded error message
- customer ID unless explicitly bounded and approved

## Logging Standards

Never log private keys, mnemonics, seed phrases, API keys, auth tokens, raw
Authorization headers, signing payloads, or unredacted RPC URLs.

Structured log requirements:

- Include `timestamp`, `level`, `service`, `environment`, `cluster`, and `correlation_id`.
- Include `program_id`, `instruction`, `endpoint_alias`, and `error_class` when relevant.
- Truncate or hash transaction signatures and wallet addresses unless needed for a
  short-lived incident investigation log with restricted access.
- Redact nested keys matching `authorization`, `apiKey`, `api_key`, `token`,
  `secret`, `privateKey`, `private_key`, `mnemonic`, `seed`, `rpc_url`.
- Workers may emit structured JSON through `console.log` because Logpush captures
  it; Node services must use a structured logger such as Pino.

## Distributed Tracing Standards

Every user-impacting transaction flow should produce one trace from frontend or
API request through RPC submission and confirmation.

Required span attributes:

- `solana.cluster`
- `solana.program_id`
- `solana.instruction`
- `solana.rpc.method`
- `solana.rpc.endpoint_alias`
- `solana.commitment`
- `solana.confirmation_status`
- `error.class`

Do not put wallet addresses, tx signatures, or raw RPC URLs in span attributes.
Use span events or restricted logs for short-lived forensic references.

## Health Checks

Standard endpoint contract:

- `GET /live` — process liveness, no external dependencies
- `GET /ready` — dependency-aware readiness for accepting traffic
- `GET /healthz` — deep health including RPC, indexer, and critical accounts
- `GET /metrics` — Prometheus metrics
- `GET /health` — optional compatibility alias only

Public health endpoints must be rate-limited to 10 requests/minute/IP. Internal
kubernetes or Docker probes may be exempt only when network-restricted.

Health responses must not expose raw RPC errors, API-key URLs, secrets, or wallet
addresses. Return endpoint aliases and sanitized error classes.

## Heartbeats and Freshness

Monitor every asynchronous pipeline with a freshness metric:

- RPC slot probe: every 15s by default
- Helius/webhook receiver: event lag and delivery totals
- Indexer: latest indexed finalized slot vs network finalized slot
- DePIN nodes: heartbeat age by bounded cohort
- Cron jobs: last successful run timestamp
- Oracle feeds: last update age and source deviation

Freshness alerts should distinguish zero traffic from broken ingestion.

## Synthetic Monitoring

Synthetic checks are required for critical user flows:

- wallet connect simulation where supported
- transaction simulation for representative instructions
- read-only account state check for critical PDAs/vaults
- frontend availability and p95 load time
- status page health endpoint check

Synthetic transactions must never sign with production treasury or authority keys.
Use read-only RPC and dedicated low-value canary accounts where sending is needed.

## Alert Rules

Alert on user-impacting ratios where possible. Use absolute thresholds for slot
lag, freshness, health availability, fee payer balance, capacity, and security
state changes.

Required alert labels:

- `severity`: `p0`, `p1`, `p2`, `p3`
- `service`
- `owner`
- `slo` or `security` when applicable
- `runbook_url`
- `dashboard_url`

Required alert properties:

- A `for:` duration unless it is an immediate security/funds-at-risk alert.
- A threshold tied to an SLO, error budget, or explicit operational limit.
- A runbook with first 5-minute actions and resolution criteria.
- A dashboard link covering the same metric and time window.

Severity standard:

| Severity | Page? | Conditions |
|---|---|---|
| P0 | Yes, immediate | funds at risk, total outage, tx failure >10%, authority compromise |
| P1 | Yes, <15m | tx failure 2-10%, RPC degradation affecting users, indexer lag >5m |
| P2 | No page | elevated errors, non-critical component down, DePIN cohort degraded |
| P3 | No page | trend, capacity warning, low-urgency anomaly |

## Alert Fatigue Prevention

Before adding a new alert, verify:

- It maps to user impact, funds risk, SLO burn, or required operator action.
- It has enough duration to avoid transient Solana noise.
- It deduplicates by bounded labels only.
- It does not page for root causes when a symptom alert already pages.
- It has suppression rules for planned maintenance and known launches.
- It was tested with a synthetic threshold or staging incident.

Never page on a single RPC error, one failed transaction, average latency alone,
or a dashboard-only business metric with no operator action.

## Dashboard Requirements

Every production dashboard must declare:

- Audience: executive, operational, technical, on-call, or public
- SLOs and thresholds used
- Data sources and freshness expectations
- Owner and review cadence
- Links to runbooks for red/yellow panels
- Deploy, incident, and program upgrade annotations

On-call dashboards must answer “is user impact happening?” in 10 seconds. Public
dashboards must not expose internal endpoint names, API-key URLs, incident-only
links, or sensitive treasury details.

## Incident Response Triggers

Immediately hand off to incident response for:

- unexplained value outflow
- upgrade/mint/freeze authority anomaly
- oracle deviation >5% consumed by protocol
- tx failure rate >10% for 5 minutes
- wallet drain or frontend compromise indicators
- fake DePIN proof exploit or impossible coverage anomaly

P1 handoff for 2-10% tx failure, RPC failover failure, indexer lag >5 minutes,
or launch claim success <98%.

## Runbook Expectations

Every P0/P1 alert must link to a runbook containing:

- severity and owner
- symptoms and user impact
- dashboard and PromQL query
- first 5-minute actions
- investigation steps
- escalation path
- rollback/failover options
- communication trigger
- resolution criteria
- post-incident monitoring action

## Performance, Capacity, and Cost Monitoring

Track capacity before it becomes an incident:

- RPC request rate and quota usage by endpoint alias
- p95/p99 latency and confirmation time
- webhook backlog and delivery failures
- indexer queue depth and lag
- Grafana/Prometheus disk retention and cardinality growth
- fee payer balance and runway
- CU p95/p99 by instruction
- storage growth for logs/traces/metrics

Cost optimization must not reduce safety coverage. Reduce cardinality, retention,
or sampling before removing critical metrics.

## AI-Assisted Anomaly Detection

AI can summarize anomalies and suggest correlations, but it must not directly
page, pause programs, rotate authorities, or send public incident updates without
human approval.

AI anomaly analysis must include:

- baseline window
- current value
- confidence level
- top contributing bounded dimensions
- suggested severity
- dashboards/log queries used
- uncertainty and next validation step

## Production Safety Rules

- Use read-only RPC connections for monitoring.
- Never hardcode secrets or API-key-bearing RPC URLs in dashboards or client code.
- Never expose Helius/QuickNode keys or other private credentials in client-side or public-facing configuration.
- Never make wallet address, tx signature, or raw RPC URL a Prometheus label.
- Never publish raw exploit details on public dashboards.
- Never claim validation passed unless promtool, JSON/YAML checks, or relevant
  tests actually ran.

## Anti-Patterns

- Single-RPC monitoring
- Average latency instead of histograms/percentiles
- Alerting on causes instead of user symptoms
- Green dashboards during zero traffic
- Runbooks that say “investigate” without concrete commands
- Endpoint labels that contain secrets
- Public dashboards with internal provider names
- Unbounded in-memory event buffers
- Program upgrade detection based only on account owner
- Dashboard JSON changed manually without version control

## Review Checklist

Before merging observability changes:

- [ ] Metric names match the canonical contract or have an adapter note.
- [ ] Labels are bounded and safe.
- [ ] P0/P1 alerts have runbooks and dashboard links.
- [ ] Health endpoints follow `/live`, `/ready`, `/healthz`, `/metrics`.
- [ ] Public endpoints are rate-limited or network-restricted.
- [ ] Logs/traces redact secrets and high-cardinality identifiers.
- [ ] Dashboards show thresholds, denominator, and data freshness.
- [ ] Synthetic checks cannot spend meaningful funds.
- [ ] Deploy docs include validation commands.
- [ ] No TODOs, placeholders, or fake dashboard IDs remain.

## SLO Definition Template

When creating an SLO, include the following template in the dashboard and alert metadata:

- **SLO name:** short descriptive name (e.g., "Transaction success rate - 30d")
- **Objective:** e.g., 99.5% success over 30 days
- **SLI:** Prometheus expression used to compute numerator and denominator
- **Window:** rolling or calendar window with retention details (30d by default)
- **Error budget:** quota in minutes or percent and burn-rate thresholds
- **Owner:** team or individual responsible for the SLO
- **Runbook URL:** canonical runbook for SLO breaches

Example SLI metadata (to include in dashboard panel description):

```
SLI expression: sum(rate(solana_transaction_total{status="success"}[5m])) / sum(rate(solana_transaction_total[5m]))
Window: 30d (rolling)
Objective: 0.995
Error budget: 0.005
```

## Runbook Template (use in `/runbooks/`)

Every runbook referenced by alerts must include a machine-readable header and a human-actionable first-5-minute checklist. Example front matter and minimal structure:

```
---
title: "TX Success Rate Drop"
severity: P0
owner: ops-team@example.com
runbook_url: /runbooks/tx-success-rate-drop.md
promql: "(1 - sum(rate(solana_transaction_total{status=\"success\"}[5m])) / sum(rate(solana_transaction_total[5m]))) / 0.005"
---

# First 5 minutes
- Confirm scope: are failures global or per-instruction?
- Check fee payer runway and priority fee p95.
- Compare primary/secondary RPC providers for slot lag.
- If funds risk, trigger emergency key rotation and treasury isolation.
- If deployment caused regression, roll back to last good release.

# Investigation steps
- PromQL queries
- Links to Grafana panels
- Links to logs/traces and explorers

# Escalation
- PagerDuty: ops-oncall
- Slack: #urgent-incidents

# Resolution criteria
- Success rate > SLO for 30m with decreasing burn rate
```

## Escalation Policy (examples)

- P0: immediate page to on-call, escalate to on-call lead after 15 minutes, notify engineering director after 60 minutes.
- P1: page to on-call, if unresolved in 30 minutes escalate to on-call lead.
- P2: no page; alert assigned with 2-hour response SLAs.

Include explicit acknowledgement and mitigation steps in the alerting metadata so responders know how to mark progress.

## Tracing Sampling Strategy

- Default: sample 1-5% of traces to balance cost and observability.
- Keep 100% of error traces and traces that include `solana.transaction` with failure tags.
- Targeted tail sampling for traces with high confirmation latency or high CU usage.
- During incidents, increase sampling to 100% for affected services/windows.

Instrumentation guidelines:

- Attach SLO, cluster, program_id, and instruction to spans where safe.
- Use short-lived, redacted identifiers for sensitive fields.

## Retention and Storage Recommendations

- Metrics: local retention 15d for hot data; remote-write to object storage or long-term TSDB for 30-365d.
- Logs: 30-90d for incident-level logs, 7-30d for application logs depending on compliance.
- Traces: keep error traces indefinitely if storage allows; sample or archive older traces.

## Dashboard and Panel Metadata Standards

Every production Grafana dashboard JSON must include:

- `title`, `description`, `owner`, `audience`, `slo_links`, `review_cadence`, `data_sources`, `variables`, `sensitive` flag.
- Panels must include `panel_id`, `queries`, `thresholds`, `runbook_url`, and `links` to related logs/traces.
- Commit dashboard JSON via Git with PR review. Use folder-level permissions to restrict edits in production folders.

## CI Validation

- Add `promtool check rules` for Prometheus rule files in CI.
- Validate Grafana JSON with a schema checker or unit tests that load and parse JSON.
- Lint runbooks for required headers and ensure alert `runbook_url` resolves in docs build.

## Example PromQL (operational)

- Current failure rate (5m):
  `1 - sum(rate(solana_transaction_total{status="success"}[5m])) / sum(rate(solana_transaction_total[5m]))`
- Error budget burn rate (30d):
  `((1 - sum(increase(solana_transaction_total{status="success"}[30d])) / sum(increase(solana_transaction_total[30d]))) / 0.005)`

## Postmortem Expectations

- All P0/P1 incidents must produce a postmortem within 7 days, with owner, timeline, root cause, corrective actions, and follow-up tasks.
- Track action item completion in the runbook and link to the corresponding PRs that implement fixes.

## Governance and Review Cadence

- SLOs and critical alerts must be reviewed quarterly or after any P0 incident.
- Owners must be documented and reachable; rotate on-call ownership at least every 6 months.
- Maintain a changelog for monitoring rule changes with reviewer and rationale.
